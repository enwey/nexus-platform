# 项目问题盘点与后续开发安排

## 文档目的

本文档用于沉淀当前 `nexus-platform` 仓库的整体问题、优先级和后续开发安排，作为后续修复和迭代的执行依据。

## 当前整体判断

当前仓库已经具备平台化方向，但主干尚未稳定。现阶段的核心问题不是单一模块实现不足，而是以下三条主线都还没有完全闭环：

- 代码主线：后端与前端都存在会阻塞编译或构建的源码问题
- 业务主线：登录、上传、下载、运行链路尚未真正打通
- 工程主线：环境配置、构建入口、客户端交付和 CI 验证还没有统一收束

## 主要问题清单

### P0：编译与构建阻塞

#### 1. backend 存在明确的编译错误

- `UserController.java` 使用 Java `record` 时仍调用 `getUsername()` / `getPassword()` / `getEmail()`，会直接导致编译失败
- 多处源码中存在字符串未闭合问题，已经不只是乱码展示问题，而是源码本身损坏
- `CacheConfig.java` 中的 Redis 缓存配置写法与当前 Spring Boot 3 生态不匹配，存在较高的构建失败风险

影响：后端当前不能被视为可稳定构建模块。

#### 2. dev-portal 存在会直接阻塞构建的 Vue 源码错误

- 多个页面存在标签未闭合、引号损坏、属性截断等问题
- 乱码已经进入模板与脚本内容，破坏了页面结构和校验规则定义
- 当前 `dev-portal` 不能作为可持续迭代的前端项目继续开发，必须先进行源码级修复

影响：开发者后台当前不能被视为可靠的联调入口。

#### 3. Android 客户端尚未进入真正可构建状态

- 当前仍缺少 `gradle/wrapper/gradle-wrapper.jar`
- 虽然 Android 工程骨架已经整理，但尚未完成真实 Gradle 构建验证
- Android 端当前更接近“整理后的运行时原型”，还不是可直接交付的客户端工程

影响：Android 工程尚未具备稳定交付和验证能力。

### P0：业务主链路未闭环

#### 4. 登录与鉴权模型不成立

- 后端注册和登录直接返回 `User` 实体
- `User` 实体包含密码字段，存在敏感信息泄露风险
- 前端请求层按 Bearer Token 模式发送鉴权头，但后端并未建立真实 token 策略
- 登录页在没有真实 token 时写入 `mock_token`，属于临时伪造逻辑

影响：当前登录流程无法作为正式业务能力继续扩展。

#### 5. 游戏上传、下载、运行链路没有真正打通

- 后端保存的游戏下载地址为相对路径
- Android 下载逻辑要求绝对 URL
- 一旦进入真机、模拟器或容器网络环境，当前路径与地址模型会失效

影响：游戏从后台上传到客户端运行的主链路没有完成闭环。

### P1：跨端桥接与平台契约不一致

#### 6. iOS 同步桥接设计与 SDK 当前行为不一致

- SDK 中同步 API 在原生环境会直接走 `invokeNativeSync()`
- iOS 当前被明确为不提供同步桥接
- 因此部分 sync API 在 iOS 上实际会抛错，而不是稳定降级

影响：桥接合同、SDK 行为和平台实现之间仍然存在漂移。

#### 7. 桥接能力虽然已收束到合同，但调用语义仍不够完整

当前合同可以描述 API 是否存在，但还不足以表达以下差异：

- 是否只支持异步
- 是否只是 stub
- 是否允许 Web 环境 mock
- 是否会在特定平台抛错

影响：后续继续扩展桥接 API 时，仍然容易发生“名称对齐了，但行为没对齐”的问题。

### P1：环境模型和工程化仍需收束

#### 8. 环境配置仍然以本地硬编码为主

- backend 默认连接 `localhost:5432`、`localhost:9000`、`localhost:7351`
- dev-portal 默认请求 `http://localhost:8080/api/v1`
- 客户端和后端之间尚未建立统一的环境注入策略

影响：本地开发还可勉强工作，但真机、容器网络和预发环境会持续出现联调问题。

#### 9. CI 还没有覆盖整个平台主线

当前 CI 已覆盖合同检查、SDK、portal、backend，但仍缺：

- Android 真实构建验证
- iOS 真实构建验证
- 端到端链路验证

影响：CI 通过并不代表整个平台可用。

## 已确认的验证边界

以下问题已经通过当前仓库检查或静态代码分析确认：

- 仓库级合同检查可以通过
- `mock-sdk` 类型检查可以通过
- Android 完整性检查会因为缺少 `gradle-wrapper.jar` 失败
- `dev-portal` 构建在当前环境中受 `spawn EPERM` 影响

以下验证尚未完成：

- backend 未完成真实 Maven 构建验证
- Android 未完成真实 Gradle 构建验证
- iOS 未完成真实构建验证
- 全链路联调未完成

## 后续修改安排

### 阶段一：修复编译与源码完整性

目标：让 backend 和 dev-portal 先恢复到“源码可构建、可继续开发”的状态。

任务：

- 修复 backend 中所有语法错误、record 访问错误和损坏字符串
- 重写或修正 `CacheConfig` 以适配当前 Spring Boot 版本
- 统一后端源码编码，清理乱码进入源码的问题
- 修复 `dev-portal` 中所有 Vue 模板、脚本和校验规则错误
- 统一前端源码编码，确保页面至少可以通过基础构建

完成标准：

- backend 能通过本地构建
- dev-portal 能通过本地构建
- 不再存在源码层面的乱码破坏结构问题

### 阶段二：打通核心业务链路

目标：让“登录 -> 上传 -> 下载 -> 运行”形成最小闭环。

任务：

- 设计并落地真实鉴权模型，至少先实现 session 或 JWT 中的一种
- 后端登录和注册返回安全 DTO，而不是直接返回 `User` 实体
- 前端去掉 `mock_token`，改为对接真实登录结果
- 梳理上传接口、游戏元数据和下载地址生成方式
- 统一客户端下载地址策略，解决相对路径与绝对路径不一致问题

完成标准：

- 开发者后台可以真实登录
- 可以上传游戏包
- 客户端可以下载并尝试加载游戏资源

### 阶段三：收束平台契约与多端桥接

目标：让 SDK、Android、iOS 的桥接能力在合同和行为上保持一致。

任务：

- 重新定义桥接能力矩阵
- 补充合同字段，区分 `implemented`、`async-only`、`stub`、`unsupported`
- 为 iOS 上的同步 API 制定正式降级策略，而不是运行时直接抛错
- 检查 Android / iOS / SDK 的实际行为是否与合同一致

完成标准：

- 主要桥接 API 的行为、平台差异和降级方式全部有据可依
- 不再出现“合同写已实现，但运行时仍报错”的情况

### 阶段四：补齐工程交付能力

目标：让仓库从“原型集合”进入“可交付工程”状态。

任务：

- 补齐 Android 官方 `gradle-wrapper.jar`
- 跑通 Android `assembleDebug`
- 明确 iOS 的真正构建入口与运行方式
- 统一根级开发命令和环境变量注入策略
- 为 backend、portal、mobile 梳理更稳定的联调说明

完成标准：

- Android 可真实构建
- iOS 有明确的可执行构建入口
- 根级工作流可支持新成员快速上手

### 阶段五：补测试与 CI

目标：让平台主线具备持续验证能力。

任务：

- backend 增加 controller / service 层测试
- dev-portal 增加基础 lint / build 校验
- mock-sdk 增加平台差异测试，尤其是 sync / async 行为测试
- CI 增加 Android 静态或真实构建验证
- 条件允许时增加 iOS 构建验证

完成标准：

- CI 能覆盖核心模块
- 关键主链路变更不再完全依赖人工回归

## 建议排期

### 第 1 周

- 完成 backend 编译修复
- 完成 dev-portal 编译修复
- 完成源码编码统一和乱码清理

### 第 2 周

- 完成真实登录模型
- 打通上传、下载、运行最小链路
- 收口客户端与后端之间的地址模型

### 第 3 周

- 收束桥接能力矩阵
- 修正 iOS 同步桥接策略
- 推进 Android 构建能力补齐

### 第 4 周

- 补测试
- 补 CI
- 恢复 UI 和业务功能的持续迭代

## 推荐执行顺序

建议不要继续并行增加新功能，而是严格按以下顺序推进：

1. 先修 backend 和 dev-portal 的编译阻塞
2. 再修登录、上传、下载、运行主链路
3. 再统一桥接合同和环境模型
4. 最后补构建、测试和 CI

## 结论

当前项目已经有平台雏形，但还不适合直接进入功能扩展阶段。最优先的工作不是继续加新能力，而是先把“能编译、能登录、能上传、能下载、能运行”这条最小主干彻底拉直。只有主干稳定后，后续 Android、iOS、SDK、后台和后端的开发才会真正进入正循环。

# 平台开发者接入规范（上传、审核、版本回滚）

## 1. 文档目标

本规范用于指导开发者在 Nexus Platform 完成小游戏全流程接入：

1. 游戏包准备
2. 开发者后台上传
3. 运营审核
4. 上线发布
5. 版本更新与回滚

适用范围：Web/HTML5 导出的小游戏（含 Cocos、LayaAir、Egret、Unity WebGL、Godot、Phaser/PixiJS 等）。

## 2. 包体与目录规范

### 2.1 ZIP 必选要求

- 压缩包根目录必须包含 `index.html`
- 所有静态资源必须使用相对路径
- 文本资源统一 UTF-8 编码
- 不依赖本地 Node.js 进程或外部本地服务

示例：

```text
my-game.zip
├── index.html
├── assets/
├── js/ 或 dist/
└── manifest.json（可选）
```

### 2.2 建议要求

- 首屏资源体积可控，减少白屏等待
- 关键文案提供简体/繁体（如项目支持多语言）
- 顶部 UI 避让胶囊区域（通过 `wx.getMenuButtonBoundingClientRect()`）

## 3. 上传流程（开发者后台）

### 3.1 创建与上传

1. 登录开发者后台
2. 进入“上传游戏”页面
3. 填写游戏名称与描述
4. 上传 `.zip` 包
5. 提交后生成游戏条目与版本草稿

### 3.2 上传校验

平台会进行基础校验：

- 是否为 ZIP 文件
- ZIP 根目录是否存在 `index.html`
- 基础元数据是否齐全

校验失败则返回错误原因，开发者需修复后重新上传。

## 4. 审核流程（运营后台）

### 4.1 状态说明

建议使用以下状态流转：

- `DRAFT`：草稿（开发者可继续改）
- `PENDING_REVIEW`：待审核
- `APPROVED`：审核通过
- `REJECTED`：审核驳回
- `ONLINE`：已上线（当前生效版本）

### 4.2 提交流程

1. 开发者在“我的游戏”提交版本审核
2. 运营在审核后台查看版本与说明
3. 审核通过后版本进入可发布状态
4. 驳回时必须附带驳回原因，开发者修复后可再次提交

## 5. 发布与版本更新

### 5.1 普通更新

- 客户端可优先启动本地缓存版本
- 后台异步检查新版本
- 若有更新，按 `UpdateManager` 流程提示用户重启生效

### 5.2 强制更新

当新版本与旧版本不兼容时，开发者在发布时勾选“强制更新”。

平台行为：

1. 客户端先进行版本检查
2. 若返回强制更新，阻断旧版本启动
3. 强制下载并解压新版本后再进入游戏

## 6. 版本回滚流程

### 6.1 触发场景

- 线上版本出现严重 Bug（崩溃、黑屏、支付异常）
- 新版本与服务端接口不兼容
- 关键路径性能劣化明显

### 6.2 回滚规则

1. 只能回滚到历史“已审核通过”的稳定版本
2. 回滚操作需记录原因（审计）
3. 回滚后客户端新会话应拉取回滚后的活跃版本
4. 回滚后建议立即发起问题版本下线或修复提审

### 6.3 推荐操作步骤

1. 在开发者后台版本列表选择目标历史版本
2. 点击“回滚到此版本”并填写原因
3. 平台切换活跃版本指针
4. 验证 Android/iOS 真机启动与核心链路

## 7. 提审前检查清单

1. ZIP 根目录包含 `index.html`
2. 首屏可见且主要交互可点击
3. 文案无乱码，简繁切换正常
4. 胶囊按钮区域无遮挡
5. 断网、下载失败有提示与重试
6. 更新提示与重启生效流程可验证
7. 关键埋点与日志可定位问题

## 8. 常见问题

### Q1: 上传成功但运行时报资源 404

通常是资源路径写成了绝对路径，或构建后路径前缀错误。

### Q2: 审核通过后用户仍打开旧版本

请检查是否为普通更新策略；如为不兼容版本，请启用强制更新。

### Q3: 回滚后仍有用户异常

确认客户端缓存指针是否已切换到回滚版本，并检查是否需要清理异常缓存。
